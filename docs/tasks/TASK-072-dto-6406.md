### Контекст и мотивация

- **Основная задача приложения** — формирование качественной Swagger-спецификации для фронтенда и других потребителей API.
- В предыдущих коммитах были обновлены Zod‑схемы и DTO для отчёта 6406, добавлены новые целевые схемы (`taskDetailsSchema` и др.), удалены/уточнены устаревшие поля и enum‑значения.

### 2026-03-12: Результаты анализа кодовой базы

**Текущее состояние схем в `tasks.schema.ts`:**

На данный момент в файле сосуществуют 4 схемы задания:

| Схема | Статус | Поля | Где используется |
|-------|--------|------|------------------|
| `baseTaskSchema` | **Новая, целевая** | `branchIdsList`, `periodFrom/periodTo`, `currencyCode`, `fileType` | Пока нигде — подготовлена как база |
| `taskDetailSchema` | **Новая, целевая** | Полная деталь + `refine()`, `s3FolderId`, `operationTypesList`, `packageId` | `formatTaskDetails`, `getTaskById`, `createTask` |
| `taskSchema` | **Устаревшая** | `branchId`, `branchName`, `periodStart/End`, `currency`, `format`, `canCancel/Delete/Start`, `fileUrl`, `errorMessage` | `formatTask` (не вызывается из роутов напрямую) |
| `taskListItemSchema` | **Устаревшая** | `branchId`, `branchIds`, `branchName`, `branchNames`, `periodStart/End`, `canCancel/Delete/Start`, `packageIds` | `formatTaskListItem`, `tasksListResponseSchema` |

**Решение:**
- `taskDetailSchema` — каноническая схема DTO для задания (деталь).
- `taskListItemSchema` нужно переписать как проекцию `taskDetailSchema` (или полностью заменить на `taskDetailSchema`).
- `taskSchema` — удалить после перевода всех использований.
- `baseTaskSchema` — оставить как общую базу для `taskDetailSchema` и (при необходимости) `taskListItemSchema`.

**Расхождения DB-схемы и целевой DTO-модели:**

| DTO-поле | DB-колонка | Статус |
|----------|-----------|--------|
| `branchIdsList` | Нет (связь через `report_6406_task_branches`) | OK, формируется из JOIN |
| `periodFrom` / `periodTo` | `period_start` / `period_end` | OK, маппинг в сервисе |
| `currencyCode` | `currency` | OK, маппинг в сервисе |
| `fileType` | `format` | OK, маппинг в сервисе |
| `sourcesList` | `source` (varchar, одиночное) | Расхождение: DTO — массив чисел, DB — одна строка |
| `operationTypesList` | Нет колонки | Расхождение: поле в DTO, нет в DB |
| `packageId` | Нет колонки (связь через `report_6406_package_tasks`) | OK, формируется из JOIN |
| `s3FolderId` | Нет колонки | Расхождение: поле в DTO, нет в DB (всегда `null`) |
| `branchId` / `branchName` (DB) | Нет в целевом DTO | Устаревшие, но активны в DB — нужна миграция или deprecation |
| `fileUrl` / `errorMessage` (DB) | Нет в целевом DTO | Устаревшие — нужно решить: удалить из DB или оставить как internal |
| `lastStatusChangedAt`, `startedAt`, `completedAt` (DB) | Нет в целевом DTO | Не выводятся в DTO — используются только внутренне |

**Фильтр `operationType`:**
В `tasksListFilterSchema` присутствует поле `operationType: z.array(zIdSchema)`, но соответствующей DB-колонки нет. В `buildFilterConditions` этот фильтр пока не маппится ни на что. Требуется решение: либо добавить колонку в DB, либо убрать из фильтров.

**`task-files.service.ts` — импорт из `../../db/index.ts`:**
Файл до сих пор импортирует `db` и схемы из `../../db/index.ts` (с расширением), тогда как остальные сервисы перешли на `../../db` (без расширения). Желательно унифицировать.
