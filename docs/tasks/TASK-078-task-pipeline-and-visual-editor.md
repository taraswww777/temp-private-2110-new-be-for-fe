# TASK-078: Пайплайн заданий («после A — делай B») и визуальный редактор цепочек

---

## Краткое описание

Реализовать **последовательный пайплайн задач**: после завершения задания **A** следующим по плану становится задание **B** (и далее по цепочке). Поддержать моделирование нескольких цепочек в рамках проекта и сценарии ветвления/слияния при необходимости (минимум — линейная цепочка; расширение — DAG с явными правилами «все предшественники завершены»).

Дополнительно — **визуальный редактор** (граф/схема), где видны задачи проекта и связи «следует за», с возможностью **массово** править порядок и связи, не открывая каждую карточку отдельно.

---

## Связь с TASK-077

**TASK-077** вводит общие связи задач (Jira-подобные) и иерархию родитель — подзадача. Пайплайн — **отдельная семантика**: «рабочий порядок исполнения», а не произвольная связь.

При реализации допустимо:

- либо завести **отдельный тип связи** (например `follows` / `nextInPipeline`) поверх той же таблицы связей из TASK-077;
- либо **отдельную сущность** «ребро пайплайна» с валидацией от циклов в смысле «завершения → следующая».

Исполнитель выбирает вариант так, чтобы не дублировать противоречивые данные; в спецификации TASK-077 при необходимости добавить ссылку на этот документ.

---

## Цели

1. **Модель данных**: явное представление «после задачи X (в статусе завершено) следующая задача Y»; правила при смене статуса назад (откат завершения) — зафиксировать (например, не трогать цепочку или помечать «устаревший порядок»).
2. **API** (`taskViewerBe`): чтение цепочки для задачи/проекта, создание/удаление ребра «следует за», опционально — список «готовых к работе» (предшественники завершены).
3. **UI** (`taskViewerFe`): на деталке задачи — блок «Пайплайн»: предыдущая / следующая; предупреждение при попытке построить цикл.
4. **Визуальный редактор**: отдельная страница или режим (например для выбранного проекта): узлы = задачи, рёбра = «затем»; drag-коннекторы или построение связей кликом; сохранение через debounce или явная кнопка «Сохранить». Технологию (React Flow, Cytoscape, собственный SVG) выбрать с учётом лицензии и объёма зависимостей.

## Критерии приёмки

- [ ] Можно задать линейную цепочку A → B → C и увидеть её в API и в интерфейсе.
- [ ] Нельзя сохранить цикл в смысле пайплайна (граф предшествования по рёбрам «следует за» ацикличен) — серверная валидация обязательна.
- [ ] При завершении задачи A пользователь однозначно видит, что **следующая по пайплайну** — B (без обязательной автосмены статуса B, если это не оговорено; при необходимости — только подсветка/подсказка).
- [ ] Визуальный редактор открывается в контексте проекта, отображает задачи и рёбра пайплайна; изменения сохраняются и отражаются на деталках задач.
- [ ] Документация контракта (OpenAPI/Swagger) обновлена; тесты на валидацию циклов и CRUD рёбер на бэкенде.
- [ ] Статус задачи в `tasks-manifest.json` обновляет исполнитель по завершении.

---

## Вне объёма

- Автоматическое выполнение или интеграция с CI «переход к следующей задаче» без участия пользователя.
- Полноценный BPMN-движок и параллельные шлюзы на первой итерации (можно заложить расширение модели текстом в коде).

---

## Примечание для исполнителя

Согласовать с продуктом: при **нескольких исходах** из одной задачи (ветвление) — блокировать «все ветки обязательны» или «достаточно одной следующей». Для MVP допустима только **одна исходящая** связь «следует за» на задачу.
