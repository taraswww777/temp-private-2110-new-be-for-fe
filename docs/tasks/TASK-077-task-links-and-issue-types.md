## Краткое описание

Ввести в Task Viewer возможность **связывать задачи между собой** и **классифицировать задачи по типам работ** в духе Jira: неслитные связи («связано с», блокировки, дубликаты и т.п.), и **структурная** связь «родитель — подзадача». Опционально расширить модель **типами issue**: история, эпик, фича, баг, задача, подзадача и др. с понятной семантикой и ограничениями на уровне контракта (что к чему может быть привязано).

Цель — не полный клон Jira, а **достаточно близкая механика**, чтобы команда могла планировать и навигироваться по связанным работам в `taskViewerBe` / `taskViewerFe`.

---

## Контекст

Сейчас задача описывается метаданными (`Task` / манифест), тегами, проектом, приоритетом и статусом. Явных **связей между задачами** и **типов работ** нет. Потребность: отображать и редактировать связи в UI, сохранять их устойчиво (в том же контуре хранения, что и манифест задач), синхронизировать контракты API и OpenAPI/Swagger при наличии.

Существующая интеграция YouTrack (в т.ч. `youtrack-links.service`) — отдельный контур; **внутренние связи между задачами Task Viewer** должны быть независимыми сущностями, при необходимости в задании можно зафиксировать правило «не путать с youtrackIssueIds».

---

## Цели

1. **Неслитные (неранговые) связи**: две задачи связаны типом связи без обязательной иерархии (аналог «Linked issues» в Jira: relates to, blocks / is blocked by, duplicates, clones — точный набор типов согласовать в дизайне API, минимум: «связано с» и направленные «блокирует»).
2. **Структурная иерархия**: одна задача может быть **родителем** другой (**подзадача**); ограничить циклы и (по согласованию) глубину или «один родитель».
3. **Типы работ (issue types)**: ввести перечень типов в стиле Jira (например: Epic, Story, Feature, Task, Bug, Sub-task); хранение — по аналогии с тегами/проектами (строковый идентификатор + отображаемое имя в metadata), либо фиксированный enum в коде на первом этапе с путём расширения.
4. **Backend** (`taskViewerBe`): модель данных, валидация, CRUD связей, поля типа/родителя в задаче, экспорт в ответы списка и детали.
5. **Frontend** (`taskViewerFe`): отображение типа и связей на карточке/деталке, создание и удаление связей, выбор родителя, фильтры/подсказки по контексту проекта (минимально приемлемый UX).

## Предлагаемая модель данных (для согласования при реализации)

Исполнитель фиксирует финальный вариант в коде и в Swagger; ниже — ориентир:

| Концепция | Вариант |
|-----------|---------|
| Родитель подзадачи | Поле `parentTaskId: string \| null` на задаче (в манифесте рядом с остальными полями задачи) **или** отдельная сущность только связей с типом `parent-child` — выбрать один подход и не дублировать |
| Неслитные связи | Таблица/массив в хранилище: `{ id, sourceTaskId, targetTaskId, linkTypeId }`; направленность задаётся типом связи (например `blocks`: A → B) |
| Типы связей | Справочник metadata (как теги): `id`, `name`, `direction`: `none` \| `directed` \| `inversePair` (для пар «блокирует / заблокирована») |
| Тип работы | Поле `issueTypeId` в задаче + файл metadata типов; подзадача может требовать непустой `parentTaskId` |

**Миграция**: существующие задачи без полей — трактовать как тип по умолчанию (например `Task`) и пустые связи.

## Критерии приёмки

- [ ] Описана и реализована модель хранения связей и (при необходимости) типов связей; данные переживают перезапуск сервиса (тот же уровень персистентности, что у манифеста задач).
- [ ] API позволяет: получить задачу со связями и родителем/детьми; добавить/удалить неслитную связь; назначить/снять родителя (подзадача); изменить тип работы.
- [ ] Валидация: нельзя связать задачу с самой собой; нет циклов в иерархии родитель — потомок; связи только между задачами **одного активного проекта** (если в продукте задачи строго в разрезе проекта — уточнить и зафиксировать правило).
- [ ] В UI отображаются тип работы, родитель, дочерние задачи и список прочих связей с типом; есть действия для управления ими без ручного редактирования JSON.
- [ ] Минимальный набор типов работ в стиле Jira доступен из UI (выбор при создании/редактировании).
- [ ] Тесты на бэкенде на валидацию циклов и базовые сценарии API; при наличии e2e — дымовой сценарий в FE (по практике проекта).
- [ ] Контракты и схемы обновлены; статус задачи и даты в `tasks-manifest.json` обновляет исполнитель по завершении.

## Вне объёма (явно)

- Полный паритет с Jira (автоматизация, ранги, версии, компоненты, Agile-доски).
- Синхронизация внутренних связей с YouTrack (отдельная задача при необходимости).
- Граф связей «как в graphviz» — только если успевается; иначе оставить последующим улучшением.

## Примечание для исполнителя

Согласовать с продуктом минимальный набор **типов связей** на первую итерацию (достаточно 2–3, остальное заложить расширением справочника). При конфликте с `youtrackIssueIds` — внутренние связи не подменяют внешние идентификаторы.
