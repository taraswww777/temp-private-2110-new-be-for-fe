# TASK‑069: Рефакторинг. Унификация использования типов в сервисах


**Статус**: ✅ Выполнено
**Ветка**: `feature/TASK-069-refactoring-services`  
**Приоритет**: высокий  

---

## Краткое описание

Провести рефакторинг сервисов проекта с целью:
- унификации типов (замена «сырых» массивов на семантически верные типы из моделей);
- устранения избыточного использования `z.infer<typeof zIdSchema>` в пользу централизованного типа `ID`.

Охватывает сервисы:  
- `export.service.ts`  
- `packages.service.ts`  
- `references.service.ts`  
- `storage.service.ts`  
- `task-files.service.ts`  
- `task-status-history.service.ts`  
- `tasks.service.ts`

---

## Контекст

В текущем коде наблюдаются:
1. **Неоднородные типы** — поля/параметры типизированы примитивными массивами (`string[]`, `number[]`), хотя существуют семантически точные аналоги в моделях данных (например, `Branch['id'][]`, `ID[]`). Это снижает типобезопасность и усложняет поддержку.
2. **Избыточный `z.infer`** — многократное определение типа `ID` через `z.infer<typeof zIdSchema>` в разных сервисах:
   - создаёт дублирование;
   - усложняет изменения (требуется правка во всех местах);
   - нарушает принцип единой точки определения типа.

## Цели

- [ ] Заменить все примитивные массивы (`string[]`, `number[]` и т. п.) на семантически верные типы из `schemas/`, `db/schema/`, `types/`.
- [ ] Централизовать определение типа `ID` в одном модуле (например, `common.schema.ts` или `types/id.ts`).
- [ ] Удалить локальные определения `ID` через `z.infer` во всех сервисах.
- [ ] Обеспечить совместимость изменений с существующими DTO и Zod‑схемами.
- [ ] Сохранить функциональность сервисов после рефакторинга.

---

## Критерии приёмки

- [ ] Все поля/параметры с примитивными массивами заменены на семантически корректные типы (например, `string[]` → `Branch['id'][]`).
- [ ] Тип `ID` определён централизованно (в одном модуле), локальные `z.infer` удалены.
- [ ] Нет ошибок компиляции после изменений.
- [ ] Тесты (если есть) проходят без изменений.
- [ ] Код сохраняет функциональность.
- [ ] Обновлены комментарии к типам (при необходимости).
- [ ] Изменения не ломают совместимость с API (DTO, схемы Zod).
- [ ] В коде отсутствуют циклические зависимости между сервисами и типами.

## Технические заметки


1. **Централизация `ID`**:
   - Определить `export type ID = string;` (или другой базовый тип) в `common.schema.ts`/`types/id.ts`.
   - Импортировать `ID` во всех сервисах из централизованного модуля.
   - Удалить локальные `export type ID = z.infer<typeof zIdSchema>;`.

2. **Замена типов массивов**:
   - Использовать типы из моделей (например, `Branch['id'][]`, `TaskStatus[]`, `ID[]`).
   - При необходимости применять приведение типов (`as Branch['id']`).


3. **Проверка инициализации**:
   - Убедиться, что типы инициализаторов соответствуют объявленным типам полей.
   - Проверить совместимость с существующими DTO и схемами Zod.


4. **Инструменты**:
   - Использовать **ESLint** для проверки типобезопасности и соблюдения стандартов кода.
   - *Prettier не используется — форматирование выполняется вручную или через встроенные средства IDE.*


5. **Тестирование**:
   - Запустить тесты после внесения изменений.
   - Проверить работу API (если доступно).
